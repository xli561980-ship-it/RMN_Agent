# 作品集定位说明

这份文档说明如何把 RMN Agent 作为 AI Agent / RAG / 技术方案表达类作品集项目来介绍。重点是把项目中的真实实现讲清楚，不把本地 Demo 说成生产系统。

## 项目定位

RMN Agent 的核心定位是：面向科研文档和实验室 SOP / 手册的双路 RAG 助手。它把论文证据和操作规范分开入库、分开检索、分开解释，用来降低“把论文里的探索性参数误当成本地批准流程”的风险。

在作品集或面试中，可以把它介绍为一个小而完整的技术原型：有明确问题背景，有可运行的入库与问答流程，有可解释的检索路由，有 Streamlit UI，也诚实说明了当前限制。

## 能力映射

| 能力 | RMN Agent 的体现方式 |
|---|---|
| 业务问题理解 | 项目聚焦实验室知识工作中的真实边界：论文能提供研究证据，但 SOP / 手册才约束操作流程。 |
| RAG 方案设计 | `ingest.py`、`rag_core.py` 和 Chroma 元数据过滤共同实现 paper / SOP 双路检索。 |
| Agent 工作流拆解 | `query_analyzer.py` 先分析问题，再决定 `PAPER_ONLY`、`SOP_ONLY` 或 `HYBRID` 路径。 |
| 文档工程能力 | 入库流程支持 PDF、`.docx`、补充材料识别、chunk 切分、embedding 和处理记录。 |
| 可演示原型构建 | `app.py` 提供聊天式 UI、入库按钮、论文范围选择、引用展示和 debug 面板。 |
| 风险边界表达 | Prompt、citation validation 和文档都明确：论文参数不能替代本地批准 SOP。 |
| 技术沟通能力 | README、架构文档和演示脚本能把实现模块、用户价值、限制和后续路线连起来说明。 |

## 面试中如何介绍

先讲问题：

“实验室和 R&D 团队的信息常常分散在论文、补充材料、设备手册和 SOP 中。普通 RAG 如果把这些文档混在一起，很容易模糊研究证据和可执行操作规范的区别。RMN Agent 是我做的一个本地 RAG 原型，用来展示如何在系统设计上处理这个边界。”

再讲方案：

“系统把论文和手册以不同 `doc_type` 写入本地 Chroma。用户提问后，query analyzer 判断问题偏论文、偏 SOP，还是需要混合检索。RAG core 再从对应路径召回证据，构造带 `citation_hint` 的上下文，调用 LLM 生成回答，并用 citation validator 做轻量引用检查。”

然后讲演示：

“我可以现场放入一组非敏感文档，运行入库，先问一个论文型问题，再问一个 SOP / 手册型问题，最后问一个 hybrid 问题。debug 面板会展示路由、检索数量和引用校验，能让评审看到系统不是黑箱回答。”

最后讲边界：

“当前版本是本地 Demo，不包含权限、审计、监控、企业集成或完整事实验证。它适合展示架构设计和原型构建能力，生产化还需要进一步工程化。”

## 演示讲解顺序

1. 背景：实验室文档多，来源类型不同，直接混合检索有风险。
2. 设计：论文和 SOP / 手册在入库时写入不同 `doc_type`。
3. 路由：query analysis 决定 paper、SOP 或 hybrid 路径。
4. 检索：Chroma 按元数据过滤召回证据片段。
5. 生成：Prompt 要求基于检索证据回答，并保留引用线索。
6. 检查：citation validation 做轻量引用和数字声明检查。
7. 演示：Streamlit UI 展示回答、引用和 debug 信息。
8. 限制：说明它是本地原型，不是生产实验决策系统。

## 不要过度宣称

不要把当前仓库说成：

- 可直接投入生产的实验室合规系统。
- 已验证的安全、医疗或实验决策工具。
- 完整企业平台。
- 已内置认证、审计、监控、CRM 集成或 Docker 部署的系统。
- 包含真实客户数据、真实 SOP 或生产密钥的系统。
- 能自动判断所有实验操作是否安全可行的系统。

更准确的说法是：

- 一个可运行的本地 RAG Demo。
- 一个展示论文证据与 SOP / 手册规范分离的作品集项目。
- 一个体现 query analysis、metadata retrieval、Prompt 编排和引用检查的技术原型。
- 一个可以继续扩展到权限、部署、监控、评估和企业知识库接入的起点。

## 回答示例

**问：为什么这个项目适合作品集？**

答：

“它不是泛泛做一个聊天机器人，而是围绕实验室文档问答中的一个具体风险设计：论文和 SOP 不能混为一谈。代码里有入库、问题分析、双路检索、Prompt 生成、引用检查和 Streamlit 演示界面。它能展示 RAG 系统设计、Agent 流程拆解、风险边界意识和技术方案表达能力。”

**问：这个项目最有辨识度的地方是什么？**

答：

“最核心的是 paper path 和 SOP path 的分离。论文提供研究证据，SOP / 手册提供操作约束。系统通过 `doc_type`、query analysis 和 Prompt 规则把这个区别贯穿到入库、检索和回答中。”

**问：当前最主要的不足是什么？**

答：

“当前是本地 Demo。它没有权限、审计、监控和部署打包；citation validation 也只是轻量规则检查。要用于真实团队，还需要更完整的数据治理、评估和工程化。”

## 使用建议

- 让 `README.md` 作为主入口，先讲项目本身，再讲作品集价值。
- 用 `docs/architecture.md` 和技术面试官讨论模块关系、数据流和设计取舍。
- 用 `docs/demo_script.md` 准备 2 分钟项目展示。
- 演示前准备一组非敏感本地语料，至少包含一篇论文和一份 SOP / 手册。
- 提到测试和评估脚本时要准确：当前是基础测试和烟测，不是完整质量评估平台。
