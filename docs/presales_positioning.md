# Presales Positioning

这份文档说明 RMN Agent 如何作为 Presales Intern / AI Solution Intern 作品集项目来呈现。目标是把仓库中的真实实现能力和岗位能力对应起来，同时避免把本地 Demo 夸大成生产系统。

## 岗位能力映射

| 售前 / AI Solution 能力 | RMN Agent 的体现方式 |
| --- | --- |
| 业务痛点识别 | 项目聚焦真实技术知识工作问题：用户需要快速、可追溯地查询论文和 SOP / 手册文档。 |
| 方案设计表达 | README 和架构文档说明了为什么需要文档分流、意图路由、检索、生成和引用校验。 |
| Demo 讲解能力 | Streamlit UI 支持短时间现场演示：入库、聊天、引用、query analysis 和 citation validation。 |
| AI Agent 实现意识 | 代码体现了实际 RAG 流程：query analysis、metadata filter、Chroma retrieval、prompt composition 和模型调用。 |
| 风险与边界表达 | 项目明确说明论文参数不能替代批准 SOP，也明确不是生产级系统。 |
| 技术到业务价值转译 | 功能被解释为更快检索、更安全的操作边界、更强可追溯性和更好的 Demo 可用性。 |
| 文档化与交付意识 | README、架构说明、演示脚本、环境变量模板、测试和 eval 脚本共同形成可评审材料。 |

## 面试中如何介绍这个项目

先讲业务问题：

“在技术团队里，信息经常分散在论文、设备手册和内部 SOP 中。普通聊天机器人虽然能快速回答，但可能会把探索性的论文内容和已批准的操作流程混在一起。我做 RMN Agent，是为了展示一个能区分知识来源并提供可追溯回答的 AI 解决方案 Demo。”

再讲解决方案：

“系统会把论文和手册以不同 metadata 写入本地 Chroma 向量库。用户提问后，query analyzer 判断问题是偏论文、偏 SOP，还是需要混合检索。RAG core 再从对应路径检索证据，构造融合 prompt，调用 LLM 生成回答，并对回答中的引用提示做校验。”

然后讲演示价值：

“在售前演示中，我可以先问一个论文总结问题，展示 paper retrieval；再问一个复现实验或安全相关问题，展示 SOP alignment。debug 面板能解释 Agent 的行为，这对客户演示和技术面试都很重要。”

最后说明当前状态：

“这是一个本地原型，重点是展示架构和可行性。真正进入客户 pilot 前，还需要认证、权限、部署、监控、评估和企业数据连接器。”

## 售前讲解话术

和 reviewer 沟通时可以按这个顺序讲：

1. **客户背景**：“客户有大量论文和 SOP，用户需要花很多时间搜索和理解文档。”
2. **核心痛点**：“风险点在于研究证据和批准操作流程不是一回事。”
3. **方案思路**：“使用一个带文档分流和意图路由的 AI Agent。”
4. **Demo 证明**：“这里是 Streamlit UI，可以入库文档、提问、查看引用和 debug 信息。”
5. **可信度故事**：“系统保留 citation hints，并检查生成回答中的引用是否来自本轮检索结果。”
6. **未来路线**：“生产化需要补充权限控制、监控、部署打包、企业系统集成和更完整评估。”

## 不要过度宣称

不要把当前仓库说成：

- 生产级实验室合规系统。
- 已验证的安全或医疗决策工具。
- 完整企业平台。
- 已内置认证、CRM 集成、监控或 Docker 部署的系统。
- 包含真实客户数据或生产密钥的系统。

更准确的说法是：

- 一个可运行的本地 AI Agent Demo。
- 一个展示 RAG 架构和售前表达能力的作品集项目。
- 一个可以继续向生产化扩展的原型。
- 一个体现知识源分离、可追溯性和 Demo 交付意识的项目。

## 面试回答示例

**问题：为什么这个项目和售前实习相关？**

回答：

“售前不只是写代码，而是理解客户工作流，并把技术转化成可信的方案故事。RMN Agent 展示了这个过程。客户问题是实验室技术文档过载，解决方案是一个双路 RAG Agent：它把论文和 SOP 分开处理，根据用户问题做意图路由，检索证据，并生成带引用的回答。我可以现场演示、解释架构、说明业务价值，同时也清楚哪些部分仍是后续规划。这个组合正好对应 Presales 或 AI Solution Intern 需要的能力。”

## 作品集使用建议

- 让 `README.md` 作为招聘官和 reviewer 的主入口。
- 面试前用 `docs/demo_script.md` 排练 30 秒和 2 分钟讲解。
- 和技术面试官沟通时重点使用 `docs/architecture.md`。
- 提到 tests 和 eval scripts，体现交付意识；同时说明更完整的集成测试仍是 future work。
- 现场演示前准备一小组非敏感本地语料，因为仓库本身不提交真实客户文档。
